# IOS_FULL_PREMISSION — Impact of "Allow Full Access" on LimeIME (iOS)

Status: INVESTIGATION (corrected April 2026 after empirical test)
Scope: iOS only. Two targets analyzed separately:
- **LimeKeyboard** — the keyboard extension (`LimeIME-iOS/LimeKeyboard/`)
- **LimeSettings** — the container app (`LimeIME-iOS/LimeSettings/`)

> **Important — common misconception corrected.** App Group container access (`FileManager.containerURL(forSecurityApplicationGroupIdentifier:)`) and shared `UserDefaults(suiteName: "group.…")` **do NOT require Full Access**. Both surfaces are gated only by the **App Group entitlement**, which both LimeKeyboard and LimeSettings already declare (`group.net.toload.limeime`). Empirical confirmation in this repo: the keyboard opens `lime.db` and produces Chinese candidates with Full Access turned **off**.

DB / layout policy: read-only investigation. No changes to `Database/*.limedb`, the runtime `lime.db`, or any layout JSON.

---

## 1. What "Allow Full Access" actually toggles

iOS keyboard extensions run in a hardened sandbox. The `RequestsOpenAccess` entitlement (declared in [`LimeIME-iOS/LimeKeyboard/Info.plist`](../LimeIME-iOS/LimeKeyboard/Info.plist) line 33, generated by [`project.yml`](../LimeIME-iOS/project.yml) line 115) advertises that the extension **wants** Full Access. The user-facing **Settings → General → Keyboard → Keyboards → LimeIME → Allow Full Access** switch decides whether iOS actually grants it.

### 1.1 Capabilities Full Access **DOES** gate

| Capability | OFF | ON |
| --- | --- | --- |
| Network (`URLSession`, sockets) | ❌ | ✅ |
| General `UIPasteboard.general` | ❌ | ✅ |
| iCloud / CloudKit | ❌ | ✅ |
| Shared **Keychain access groups** | ❌ | ✅ |
| Location, Microphone, Camera, Contacts, Photos, HealthKit | ❌ | ✅ (still requires the per-API permission prompt) |
| Some `NSExtensionContext` IPC | ❌ | ✅ |

### 1.2 Capabilities Full Access **does NOT** gate

These work in either state, as long as the extension declares the matching entitlement:

| Capability | Why it works without Full Access |
| --- | --- |
| **App Group container** (`FileManager.containerURL(forSecurityApplicationGroupIdentifier:)`) | Gated by the App Group entitlement on both targets — declared in [`LimeKeyboard.entitlements`](../LimeIME-iOS/LimeKeyboard/LimeKeyboard.entitlements). |
| **Shared `UserDefaults(suiteName: "group.net.toload.limeime")`** | Same — uses the App Group, not Full Access. |
| Read/write the extension's own sandbox files | Always allowed. |
| Read the extension's **bundle resources** (layout JSONs, assets) | Always allowed. |
| Extension's **private** `UserDefaults.standard` | Always allowed. |
| `UITextDocumentProxy` (text before/after cursor, insert, delete) | Always allowed. |
| `advanceToNextInputMode()`, `dismissKeyboard()`, `handleInputModeList(...)` | Always allowed. |
| Memory budget (~48 MB hard cap) | Unchanged either way. |
| Drawing / touch handling inside the keyboard view | Unchanged either way. |

### 1.3 Things Full Access **never** unlocks

- Drawing outside the keyboard view (no overlay above the host app).
- Intercepting secure text fields (passwords) — iOS forces the system keyboard there.
- Accessing the iPad **input assistant bar** (see [IPAD_KEYBOARD.md §6](IPAD_KEYBOARD.md)).
- Running in the background when the keyboard is not shown.

### 1.4 The system warning

iOS shows a one-time prompt when the user toggles Full Access on:

> *"Full Access allows the developer of this keyboard to transmit anything you type, including things you have previously typed with this keyboard. This could include sensitive information such as your credit card number or street address."*

There is no programmatic way to request Full Access — only the user can flip the switch in Settings.

## 2. SCOPE A — LimeKeyboard (extension) WITHOUT Full Access

App Group identifier: `group.net.toload.limeime` (declared in [`LimeKeyboard.entitlements`](../LimeIME-iOS/LimeKeyboard/LimeKeyboard.entitlements) and matched on the LimeSettings side).

### 2.1 App Group container access — WORKS

Code path: [`KeyboardViewController.setupDatabase()`](../LimeIME-iOS/LimeKeyboard/KeyboardViewController.swift) (≈L343–L348)

```
guard let containerURL = FileManager.default.containerURL(
        forSecurityApplicationGroupIdentifier: "group.net.toload.limeime")
else { return }
let dbPath = containerURL.appendingPathComponent("lime.db").path
```

Succeeds without Full Access — the App Group entitlement alone is sufficient. `lime.db` opens, IM tables are queried, candidates are produced.

User-visible result: **Chinese IM input works.** Phonetic / Array / CJ / Dayi / ET26 / ET41 / Hsu / EZ / HS / WB all produce candidates as expected.

### 2.2 Shared `UserDefaults(suiteName:)` — WORKS

Code path: [`LIMEPreferenceManager`](../LimeIME-iOS/Shared/Preferences/LIMEPreferenceManager.swift) (≈L21) plus all read sites in `KeyboardViewController.swift` — for example `number_row_in_english` (≈L150), `lime_db_restored_at` (≈L177), `keyboard_state` (≈L365), `keyboard_list` (≈L392), `loadSettings()` (≈L556+).

Same reasoning — shared `UserDefaults` is gated by the App Group, not Full Access.

User-visible result: **Theme, IM selection, key size, font size, learn-phrase pref all read correctly.** Customizations made in LimeSettings take effect on the keyboard.

### 2.3 Phrase / related-phrase learning writebacks — WORK

Triggers: [`KeyboardViewController.swift`](../LimeIME-iOS/LimeKeyboard/KeyboardViewController.swift) ≈L1597–L1619 (`addLDPhrase`, `learnRelatedPhraseAndUpdateScore`).
Writers: [`SearchServer.swift`](../LimeIME-iOS/Shared/Search/SearchServer.swift) ≈L525–L578 / L691–L702 (`addOrUpdateMappingRecord`, `updateScore`, `addOrUpdateRelatedPhraseRecord`).

Because §2.1 holds, the DB is open read/write and these writes complete normally.

User-visible result: **User dictionary grows, scores promote, related-phrase pairs accumulate** even without Full Access.

### 2.4 Heartbeat / status reporting back to LimeSettings — WORKS

Code path: [`KeyboardViewController.swift`](../LimeIME-iOS/LimeKeyboard/KeyboardViewController.swift) ≈L160–L162 and ≈L185:

```
sharedDefaults?.set(true, forKey: "keyboard_extension_loaded")
sharedDefaults?.set(hasFullAccess, forKey: "keyboard_has_full_access")
sharedDefaults?.set(Date().timeIntervalSince1970, forKey: "keyboard_last_seen_at")
```

`sharedDefaults` is the shared suite handle, which works without Full Access. The `hasFullAccess` value being written is the actual `UIInputViewController.hasFullAccess` flag — `false` when the user has not granted Full Access.

User-visible result: LimeSettings's Setup tab reads these keys and shows the correct status banner (orange ⚠ when Full Access is off, green ✓ when on). The banner is **informational** — it does not reflect any current functional regression in the keyboard.

### 2.5 What actually breaks without Full Access

Of the LimeKeyboard code paths that exist today, only the following are blocked when Full Access is off:

| Capability | Used by LimeKeyboard today? | Impact when off |
| --- | --- | --- |
| Network (`URLSession`) | **No** — extension is fully offline. | None today. Future cloud sync / telemetry would fail. |
| `UIPasteboard.general` | **No** — no clipboard paste/copy feature on iOS. | None today. Future clipboard key would fail. |
| Shared Keychain access group | **No.** | None today. |
| iCloud / CloudKit | **No.** | None today. |
| Microphone (e.g. dictation key) | **No.** | None today. |

In other words, **LimeKeyboard has no current feature that depends on Full Access.** Empirically, the keyboard is fully functional without it.

### 2.6 Bundled layout JSONs — UNAFFECTED

[`LayoutLoader`](../LimeIME-iOS/LimeKeyboard/LayoutLoader.swift) loads layouts (`lime_english.json`, `symbols1.json`, `phone.json`, future `lime_*_ipad.json`, …) from `Bundle.main` of the extension. Bundle access never required Full Access.

### 2.7 Summary table — LimeKeyboard without Full Access

| Feature | Status | Notes |
| --- | --- | --- |
| English QWERTY typing | ✅ Works | Bundled layouts. |
| Symbols / numpad / iPad layouts | ✅ Works | Bundled layouts. |
| Globe key (system IM switch) | ✅ Works | UIKit call, ungated. |
| Chinese IM input (every IM) | ✅ Works | App Group access ungated. |
| Active IM selection / `keyboard_list` | ✅ Works | Shared `UserDefaults` ungated. |
| Theme / key size / font size | ✅ Works | Same. |
| Learn-phrase / score updates | ✅ Works | DB is open R/W. |
| Related-phrase learning | ✅ Works | Same. |
| Heartbeat to LimeSettings | ✅ Works | Reports `hasFullAccess = false` correctly. |
| Network sync / telemetry | ❌ Blocked | Not used today — no regression. |
| `UIPasteboard.general` | ❌ Blocked | Not used today — no regression. |

**Net user experience: no functional regression today.** LimeIME works as a Chinese IM with full learning, full customization, and full layout support without Full Access.

## 3. SCOPE B — LimeSettings (container app) — independent of keyboard's Full Access

LimeSettings is a normal app target. Its own behavior is **completely independent** of whether the keyboard extension has Full Access — the container app already has direct file/UserDefaults access to its own App Group container.

### 3.1 What LimeSettings does

Code: [`MainViewController`](../LimeIME-iOS/LimeSettings/MainViewController.swift), [`IMCatalog`](../LimeIME-iOS/LimeSettings/IMCatalog.swift), [`LimeSettingsView`](../LimeIME-iOS/LimeSettings/LimeSettingsView.swift), [`Views/SetupTabView`](../LimeIME-iOS/LimeSettings/Views/SetupTabView.swift).

- **First-launch DB import**: copies the bundled `Database/array.limedb` / `array10.limedb` seeds into the App Group container as `lime.db`.
- **Pref editing**: writes to shared `UserDefaults(suiteName: "group.net.toload.limeime")`.
- **IM management**: enables/disables IMs, manages user dictionary records inside `lime.db`.
- **Setup guidance**: shows a status banner instructing the user to (1) enable LimeIME in system Settings, then (2) **Allow Full Access** (precautionary; see §3.3).

All of these work normally regardless of the keyboard's Full Access state.

### 3.2 Full Access detection in LimeSettings

Code path: [`Views/SetupTabView`](../LimeIME-iOS/LimeSettings/Views/SetupTabView.swift) ≈L72–L79 (workflow comment), state machine ≈L177–L185, banner ≈L285–L311.

The Setup tab decides which banner to show by reading `keyboard_has_full_access` from the shared suite (the value the extension wrote in §2.4):

```
fullyEnabled        → green ✓   (keyboard enabled AND keyboard_has_full_access == true)
enabledNoFullAccess → orange ⚠  (keyboard enabled, flag false or absent)
notEnabled          → red ✗     (keyboard not in the system list)
```

Detection mechanic: SetupTabView mounts an invisible probe text field, focuses it briefly so the keyboard loads and writes the heartbeat, then defocuses it. Polled at 1-second intervals. When the user toggles Full Access in system Settings and returns to LimeSettings, the banner updates within ~1–2 seconds without leaving the screen.

### 3.3 Is the orange "Full Access not granted" banner actually justified?

**Today: largely no.** Given §2, the keyboard is fully functional without Full Access. The banner is precautionary — it covers two cases:

1. **Future-proofing**: any future feature that needs network (cloud sync, dictionary updates, telemetry), system clipboard (paste key), or microphone (dictation) will require Full Access.
2. **HIG convention**: Apple's HIG suggests keyboard apps prompt for Full Access during onboarding because most third-party keyboards do need it. LimeIME follows the convention even though its current feature set does not strictly require it.

A future cleanup option (not implemented here) would be to soften the banner copy from "required" to "optional, enables future features" — see §5.

### 3.4 Summary table — LimeSettings (regardless of keyboard's Full Access state)

| Feature | Status | Notes |
| --- | --- | --- |
| App launches | ✅ | Regular app target. |
| First-launch DB import (seeds → `lime.db` in App Group) | ✅ | Container app has direct App Group access. |
| Pref editing (theme, IM list, sizes) | ✅ | Writes to shared `UserDefaults`. |
| IM management / user dictionary edits | ✅ | Writes to `lime.db` in App Group. |
| Setup-tab Full Access detection | ✅ | Polls `keyboard_has_full_access`. Correct in all states. |
| Onboarding banner | ✅ | Precautionary — see §3.3. |

**LimeSettings has no functional dependency on the keyboard's Full Access switch.**

```
│  • Normal app sandbox   │      App Group:           │  • Hardened ext sandbox      │

│                         │      • shared UserDefaults│                              │
└─────────────────────────┘                           └──────────────────────────────┘

                                              network · UIPasteboard · iCloud ·
                                              Keychain group · location · mic · …
```

**The App Group entitlement is the channel.** Full Access is a separate, orthogonal switch that gates a different set of capabilities (none of which LimeIME uses today).

## 5. Sensitive-app exclusion is independent of Full Access

Three **OS-level** mechanisms can block 3rd-party keyboards entirely. **None of them check Full Access** — Full Access status changes nothing in any of these cases.

| Mechanism | What it blocks | Effect on LimeIME |
| --- | --- | --- |
| `UITextField.isSecureTextEntry = true` (password / PIN fields) | iOS forcibly swaps in the system keyboard for that field | LimeIME never receives the field. Same with or without Full Access. |
| Host app sets `UIApplicationSupportsExtensionPointIdentifiers` in `Info.plist` excluding `com.apple.keyboard-service` (most banking apps, some password managers, some MDM apps) | All 3rd-party keyboards blocked app-wide | LimeIME is not even loaded for that app. Same with or without Full Access. |
| MDM / Configuration Profile `allowKeyboardExtensions = false` | All 3rd-party keyboards hidden from system list | LimeIME is not selectable. Same with or without Full Access. |

Users who worry "my banking app might leak data through LimeIME" are protected by iOS itself, not by anything LimeIME does or doesn't request. Granting or withholding Full Access does not influence this layer.

## 6. "No network" policy clarification — applies to LimeIME's own code only

LimeIME's privacy promise is **"we never send anything to the internet."** This applies to **LimeIME-authored code only**:

- ❌ No `URLSession` calls in our codebase.
- ❌ No CloudKit / iCloud usage.
- ❌ No telemetry, analytics, or crash reporters.
- ❌ No cloud sync of user dictionary.
- ❌ No online phrase / theme / emoji downloads.

**Apple-provided system services that happen to use the network are NOT counted as LimeIME network access**, because:

1. The user has a separate relationship with Apple (already accepted Apple's privacy policy at device setup).
2. The data path is `user → Apple → Apple's servers`, never touching LimeIME's process or any LimeIME-controlled server.
3. Apple's services are subject to Apple's published privacy guarantees, not ours.

Examples of Apple-provided services LimeIME may legitimately invoke under the no-network rule:
- **Apple's dictation** (`SFSpeechRecognizer`) — see §7. We will further restrict to **on-device mode only** so even Apple's servers are never contacted, but the *policy* would permit either mode.
- The **system QuickType / OTP-autofill** strip — LimeIME doesn't render or suppress it; iOS hides it for 3rd-party keyboards anyway.
- System share sheet / `UIActivityViewController` if ever invoked from the container app.

This distinction is important for §7: invoking Apple's dictation does not violate LimeIME's no-network policy — LimeIME itself sends nothing, it just hands an audio buffer to a system framework.

## 7. Voice input — implementation plan (requires Full Access)

### 7.1 Why we need to ship this ourselves on most devices

The system-level dictation entry points:

| Device | OS | Entry point | Available while LimeIME is active? |
| --- | --- | --- | --- |
| iPhone 14 Pro / 15 / 16 series | 17+ | Dynamic Island mic chip (independent of keyboard) | ✅ Yes |
| iPhone 13 / SE 3 / older | any | Apple-keyboard mic only | ❌ No — disappears when LimeIME is the active keyboard |
| All iPads (any year, on-screen keyboard) | any | Apple-keyboard mic only | ❌ No |
| iPad with Magic / Smart Keyboard (hardware kbd) | 14+ | Hardware Fn / 🎤 key | ✅ Yes |

So **on every device except iPhone 14 Pro and newer (or an iPad with a hardware keyboard), users have no way to invoke voice input while LimeIME is active** unless they globe-switch to Apple's keyboard each time. To close that gap we add our own mic key.

### 7.2 API surface

Use Apple's `Speech` framework (`SFSpeechRecognizer`). Two modes:

| Mode | Audio leaves device? | Languages | Hardware requirement |
| --- | --- | --- | --- |
| Server-based (default) | **YES — sent to Apple's servers** | All supported locales | A8+ |
| **On-device** (`requiresOnDeviceRecognition = true`, iOS 13+) | **NO** | Limited subset; zh-TW / zh-HK / zh-CN / en-US / en-GB / ja-JP all supported on A12+ | **A12 Bionic or newer** (iPhone XS / XR / 11 / 12 / 13 / 14 / 15 / 16; iPad mini 5+, iPad Air 3+, iPad 8th gen+, iPad Pro 2018+) |

**LimeIME must use on-device mode only.** Per §6 server-based mode would be technically compatible with our policy (Apple-mediated), but voice audio is unambiguously sensitive content; forcing on-device keeps the privacy story unambiguous.

```swift
let request = SFSpeechAudioBufferRecognitionRequest()
request.requiresOnDeviceRecognition = true   // ← MUST be true; refuse if unavailable
request.shouldReportPartialResults = true
```

### 7.3 Why this REQUIRES Full Access

All four sub-capabilities below are gated by `RequestsOpenAccess`:

| Sub-capability | Gated by Full Access? |
| --- | --- |
| Microphone hardware access from a keyboard extension | ✅ Yes |
| `NSMicrophoneUsageDescription` prompt firing inside an extension | ✅ Yes |
| `SFSpeechRecognizer` instantiation inside an extension | ✅ Yes (`Speech.framework` refuses authorization without Full Access) |
| `NSSpeechRecognitionUsageDescription` prompt | ✅ Yes |

**Consequence**: shipping the mic key flips the answer to "is Full Access necessary for LimeIME?" from NO to YES. The orange ⚠ banner in LimeSettings becomes accurate again the day this feature ships.

### 7.4 Required Info.plist additions (extension target)

In [`LimeIME-iOS/LimeKeyboard/Info.plist`](../LimeIME-iOS/LimeKeyboard/Info.plist):

```
<key>NSMicrophoneUsageDescription</key>
<string>萊姆輸入法需要麥克風以提供離線語音輸入；錄音不會離開您的裝置。</string>

<key>NSSpeechRecognitionUsageDescription</key>
<string>萊姆輸入法以裝置上的語音辨識將語音轉為文字；不會傳送音訊或文字至雲端。</string>
```

Both prompts fire **once** on first mic-key tap. If denied, mic key shows a disabled state with a one-line tooltip pointing to Settings.

### 7.5 UI placement & gestures

- **Bottom row**: mic key already present in the iPad scaffold ([IPAD_KEYBOARD.md §4.2 bottom-row template](IPAD_KEYBOARD.md)). On phone, add a 🎤 SF-symbol key between the globe and `.?123` — width ≈ 8% (one alpha-cell width).
- **Visual states**:
  - Idle: `mic`, normal modifier color.
  - Listening: `mic.fill`, accent color; candidate bar shows live partial transcript in italic gray.
  - Disabled / unavailable: `mic.slash`, disabled color, tap surfaces a status message.
- **Gestures**:
  - Tap → toggle listening on/off (Apple convention).
  - Long-press → push-to-talk: hold = listen, release = commit & stop.
- **Output path**: partial results stream into `CandidateBarView` (replaces composing display, dim styling). On final result or `.endAudio` → standard `insertText` via `delegate?.keyboardView(self, didPress:)`. User can tap any visible IM candidate to override.

### 7.6 Hardware gating (no silent fallback)

```swift
guard SFSpeechRecognizer(locale: Locale(identifier: "zh-TW"))?
        .supportsOnDeviceRecognition == true else {
    micKey.isEnabled = false
    micKey.toolTip = "此裝置不支援離線語音辨識"
    return
}
```

Pre-A12 devices (iPhone 6s/SE1/7/8/X; iPad up to 6th gen, iPad mini 4, iPad Air 2): mic key visibly disabled. **Never silently fall back to server mode.**

### 7.7 Battery & lifecycle

- Hard 30-second session timeout.
- Stops on: `viewWillDisappear`, 1.5 s of detected silence after the first recognized token, or explicit second tap on mic key.
- Audio session: category `.record`, mode `.measurement`, options `[.duckOthers]`. Restored to `.ambient` on stop so background music resumes.

### 7.8 Locale selection

- Default to `keyboard_state`-derived locale: phonetic / array / cj / dayi / et26 / et41 / hsu / ez / hs / wb → `zh-TW`; english / abc → `en-US`.
- Long-press mic key → popup with `[zh-TW, zh-HK, zh-CN, en-US, ja-JP]` for one-shot override.
- Stored in shared `UserDefaults` key `voice_input_locale` so LimeSettings can later expose a default-language pref.

### 7.9 Failure modes

| Condition | UI |
| --- | --- |
| Full Access OFF | Mic key hidden entirely (gated on `hasFullAccess`). |
| Mic permission denied | `mic.slash` icon, tap shows alert with Settings deep-link. |
| Speech permission denied | Same. |
| Pre-A12 device (zh) | `mic.slash`, tooltip "此裝置不支援離線語音辨識". |
| Locale not on the offline whitelist | Refuse to start; "此語言尚未支援離線辨識". |
| Mic in use by another app (call, voice memo) | Apple framework returns error → key flashes red, brief inline message. |

### 7.10 What does NOT change

- **LimeSettings does not need any new permission** — never touches the mic. It can later expose a "voice input default language" pref.
- No DB schema change.
- No layout JSON change beyond optionally adding a mic-key cell on the phone bottom row (iPad already has it per [IPAD_KEYBOARD.md §4.2](IPAD_KEYBOARD.md)).
- Phrase-learning, IM lookup, candidate ranking — all unchanged.

### 7.11 Privacy copy update (when this ships)

Update the LimeSettings Setup-tab orange banner to truthfully say:

> 啟用「允許完整取用」可使用離線語音輸入。萊姆輸入法本身不會收集或傳送任何資料；語音辨識完全在您的裝置上由 Apple 系統元件執行，音訊不會離開裝置。

### 7.12 Roll-out order

1. Status quo: keep `RequestsOpenAccess = true`; mic key not yet implemented.
2. Land the on-device check (`SFSpeechRecognizer.supportsOnDeviceRecognition`) and a stub mic key that just shows an alert "Coming soon."
3. Land the actual recognition session, partial-result rendering, and commit path.
4. Add LimeSettings "voice input language" pref.
5. Update the orange banner copy.
6. Release notes: 「離線語音輸入：點擊鍵盤上的麥克風鍵直接語音輸入，音訊不離開您的裝置 (需 A12 或更新的 iPhone/iPad)。」

## 9. Verification checklist (manual, repeatable)

1. **Fresh install, Full Access OFF**:
   - LimeSettings: opens, can browse IM catalog, can toggle prefs. Setup tab shows orange ⚠ (precautionary).
   - LimeKeyboard: globe-switch into LimeIME inside any text field. **English typing works. Switching to phonetic / array / CJ → candidates appear normally.** ✅ (this is the empirical observation that corrected the original draft of this doc.)
2. **Toggle Full Access ON in system Settings, return to LimeSettings**:
   - Setup tab banner flips to green ✓ within 1–2 seconds.
   - Keyboard behavior is identical — no new functionality unlocks today (but future network / clipboard / mic features would).
3. **Toggle Full Access OFF again**:
   - Setup tab banner flips back to orange ⚠ within 1–2 seconds.
   - Keyboard behavior is again identical.
4. **App-Group sanity** (any state):
   - `lime.db` exists at `<group container>/lime.db` and is readable by both targets.
   - Shared `UserDefaults` keys (`keyboard_state`, `keyboard_list`, `keyboard_has_full_access`, `keyboard_last_seen_at`) are populated and updating.
5. **DB integrity** (regardless of state):
   - `Database/array.limedb` / `array10.limedb` byte-identical to repo (`shasum` unchanged).
   - Runtime `lime.db` schema and `keyboard` / `im` table contents unchanged after install + first-launch import.
